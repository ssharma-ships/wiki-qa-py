### Phase 3 — Frozen Retrieval Substrate (in progress)

**File 1 complete: `wikipedia_client.py`**

One public function:

```python
search_wikipedia(query: str, k: int = 3) -> dict
```

- Hits the live MediaWiki API at https://en.wikipedia.org/w/api.php
- Query params: action=query, generator=search, prop=extracts|pageprops|info,
  exintro=1, explaintext=1, ppprop=disambiguation, inprop=url, redirects=1
- User-Agent: "ClaudeWikiQA/0.1 (prompt-eng-takehome; contact@example.com)"
- Truncates each extract at 1500 chars; appends "... [truncated]" if truncated
- Sorts results by the `index` field (search rank order)
- Returns: `{"query": str, "results": [{"title", "extract", "url", "is_disambiguation"}, ...]}`
- Retry logic: one retry with 1s sleep on failure; returns `{"error": ...}` if both fail
- Zero results: returns `{"query": query, "results": []}` — no fabricated suggestions
- 100ms sleep after every successful call
- Verified working: tested with "Marie Curie Nobel Prize" and "mercury"
  - "mercury" correctly flags the disambiguation page with `is_disambiguation: true`

## Key Design Decisions

- Retrieval is fixed and not optimized — attribution stays clean
- `is_disambiguation` flag is surfaced per result so the agent loop can
  detect ambiguous queries
- No class wrapper, no logger, no caching in the client
- Tool schema (tools.py) is frozen after this phase

## Hypotheses to Validate Later

- **H1**: Baseline will rely on latent knowledge rather than grounding claims
  in retrieved Wikipedia evidence
- **H2**: Ambiguous / multi-hop questions will fail more than simple factual ones
- **H3**: System will answer even when evidence is insufficient instead of
  abstaining
