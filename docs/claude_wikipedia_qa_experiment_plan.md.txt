# Claude + Wikipedia QA Experiment Plan

---

## Phase 1 — Framing and Assumptions

### What you are doing
Build a controlled experimentation loop for Claude + Wikipedia QA.

### Core framing
Claude is the reasoning layer. Wikipedia is the evidence source. Final factual claims should be supported by retrieved Wikipedia evidence. Retrieval is intentionally simple and fixed.

### Why this matters
This prevents the assignment from being judged as a production retrieval system. The project is about prompt behavior, evaluation quality, and explainable iteration.

### Output
A short design note stating:

> I treat Wikipedia as the system’s source of evidence and Claude as the reasoning layer over that evidence. The model may use general language understanding to interpret and synthesize, but factual claims in the final answer should be supported by retrieved Wikipedia evidence. When evidence is insufficient, the system should search again, narrow the claim, or say that the evidence is insufficient.

---

## Phase 1.5 — Pre-committed Hypotheses

### What you are doing
Before building, write down a small number of hypotheses about where the system is likely to fail.

### Updated hypotheses

**H1 — Evidence-backed answering**  
The baseline system will often answer from Claude’s latent knowledge rather than grounding claims in retrieved Wikipedia evidence.

**H2 — Complex question handling**  
Ambiguous or multi-hop questions will fail more often than simple factual questions.

**H3 — Honesty under insufficient evidence**  
The system will answer even when evidence is insufficient instead of abstaining.

### Why this matters
These map cleanly to evaluation dimensions without drifting into irrelevant areas.

## Phase 2 — Scope and Non-goals

### Build
- Claude + system prompt  
- `search_wikipedia(query: str)` tool  
- Simple CLI  
- Logging  
- Eval runner  

### Do not build
- Vector DB  
- Complex ranking  
- UI  
- Multi-agent orchestration  
- Production retrieval  

### Scope statement
Retrieval is fixed and not optimized.

### Why this matters
Avoids attribution confusion.

## Phase 6 — Evaluation Rubric

### Dimensions

1. **Correctness** — factual accuracy  
2. **Evidence Support** — grounded in Wikipedia  
3. **Honesty** — avoids unsupported claims  
4. **Task Effectiveness** — answers the actual question  
5. **Answer Quality** — clarity and usefulness  

### Note
Search usage is telemetry, not a scored metric.

---

- Decompose questions

- Improve clarity and structure
